# SDD-L Principles

Spec Driven Development + Learning

## 目的

SDD-L は、AI による実装支援を活用しつつ、人間の理解・設計力・レビュー力を育てることを目的とした開発手法である。

この手法の主眼は開発効率ではない。
最終目標は、以下を満たすことである。

- 動くプロダクトを作ること
- 人間がその設計と実装を説明できること
- 人間がレビューできること
- 開発を通して知識と判断力が向上すること

## 基本思想

- 仕様は重要だが、仕様だけでは不十分である
- AI が正しいコードを書いても、人間が理解できなければ学習としては失敗である
- 人間は意思決定者であり、AI は支援者である
- AI の出力は常に「理解可能性」「レビュー可能性」を伴うべきである
- 学習者の理解負債は、技術的負債と同様に管理対象とする

## 人間の役割

人間は以下の責務を持つ。

- 何を作るかを決める
- 提案された設計案から採用方針を決める
- 実装結果をレビューする
- 分からなかった点を放置しない
- 最終的な責任を負う

## エージェント一覧

SDD-L では、以下のエージェントを利用する。

### Mentor

- 要件整理
- 設計相談
- タスク分解
- レビュー支援
- 学習者のレベルに応じた設計の調整

### Coder

- 実装
- コード編集
- 変更内容の記録
- 設計上の判断理由の明文化
- 注意点や懸念点の記録

### Teacher

- Coder の変更内容の解説
- 学習者の習熟度に応じた説明
- 理解確認
- 次に学ぶべき論点の提示

## 支援レベル

SDD-L は、学習者の習熟度やタスク領域に応じて、利用するエージェントを段階的に減らせるよう設計されている。

目標は、多くのエージェントを使い続けることではない。
より少ない支援で、人間が設計・理解・レビューを自力で行えるようになることが目標である。

### 1. Guided Mode

利用エージェント:

- Mentor
- Teacher
- Coder

対象:

- 初学者
- 不慣れな技術領域に取り組む人
- 要件整理や設計段階で詰まりやすい人

特徴:

- Mentor が設計と思考整理を支援する
- Coder が実装を担う
- Teacher が実装内容の理解を支援する
- 学習者は「何を作るべきか」「なぜその設計か」「どう読むべきか」を段階的に学ぶ

### 2. Assisted Mode

利用エージェント:

- Teacher
- Coder

対象:

- 設計の大枠は自力で考えられる人
- 実装後の理解整理や定着支援を必要とする人
- 中級レベルの学習者

特徴:

- 設計は人間が主導する
- Coder が実装を担う
- Teacher が設計意図や実装内容の理解を支援する
- Mentor なしで設計できるが、理解の補助はまだ受ける段階である

### 3. Autonomous Mode

利用エージェント:

- Coder

対象:

- 設計・レビュー・理解整理を自力で行える人
- 十分に自走可能な上級者
- ある領域に十分習熟している人

特徴:

- 人間が設計とレビューを主導する
- Coder は実装支援と変更記録を担う
- Teacher や Mentor の支援なしで、実装内容を追跡・評価できる状態を目指す

## 重要な考え方: 支援レベルは人全体ではなく領域ごとに決める

SDD-L では、学習者を一律に「初心者」「中級者」「上級者」と固定しない。

同じ人でも、領域によって習熟度は異なる。
たとえば以下のような状態は自然である。

- React では中級者
- バックエンド設計では初心者
- セキュリティレビューでは初学者

そのため、支援レベルは人全体ではなく、タスクや技術領域ごとに決める。

例:

- 慣れていない領域: Guided Mode
- ある程度扱える領域: Assisted Mode
- 十分に自走できる領域: Autonomous Mode

## 成長モデル

SDD-L は、AI 依存を強めるための手法ではない。
AI の支援を受けながら、人間が段階的に自走できるようになることを目指す。

理想的な成長は以下の流れである。

1. Mentor の助けを借りて設計を学ぶ
2. Teacher の助けを借りて実装理解を深める
3. レビュー観点を身につける
4. 徐々に支援を減らし、自分で設計・理解・レビューできるようになる
5. 最終的に Coder のみを使う、もしくはそれに近い状態に到達する

## 品質基準

どのエージェントも、以下を満たす出力を目指すこと。

- 曖昧さが少ない
- 理由が明記されている
- 学習者が追跡できる
- 人間がレビューしやすい
- 将来の修正に耐える

## 禁止事項

- 学習者の理解を置き去りにして実装だけを進めること
- なぜそうしたかの説明なしに変更すること
- 過剰に難解な設計を提案すること
- 人間の判断を奪うこと
- レビュー不能な巨大変更を一気に行うこと

## 成功条件

SDD-L における成功とは、単に機能が完成することではない。
以下を満たして初めて成功とみなす。

- 人間が変更の目的を説明できる
- 人間が主要な設計判断を説明できる
- 人間が主要な懸念点を把握している
- 人間がレビュー観点を持っている
- 残った理解不足が記録されている

## SDD-L の最終目標

SDD-L の最終目標は、AI を多く使い続けることではない。
AI の支援を減らしても、人間が設計・理解・レビューを行えるようになることである。

したがって、SDD-L は
「AIをうまく使うための手法」
であると同時に、
「AIの支援を減らしながら成長するための手法」
でもある。
